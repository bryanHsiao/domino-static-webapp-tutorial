# GitHub Actions Artifact To UpdateSite Guide

This document explains how to take the build output generated by GitHub Actions and use it to update `updatesite.nsf` on Domino.

It covers the practical path after CI is already working.

## Scope

This guide is for the current repository:

- `https://github.com/bryanHsiao/domino-static-webapp-tutorial`

Current workflow:

- workflow name: `Build Tutorial Site`
- workflow file: `.github/workflows/ci.yml`

Current artifact names:

- `update-site-repository`
- `hello-webapp-bundle`

## What CI Already Does

The GitHub Actions workflow already performs:

1. `npm ci`
2. `npm run build`
3. `mvn package`
4. uploads the generated update-site repository as an artifact
5. uploads the bundle JAR as an artifact

That means CI already produces the deployable p2 repository.

What it does **not** do yet:

- import directly into `updatesite.nsf`
- restart Domino HTTP
- activate a new server runtime automatically

So the current model is:

- automatic build
- manual Domino deployment

## Which Artifact You Need

For updating `updatesite.nsf`, the important artifact is:

- `update-site-repository`

That artifact corresponds to:

- `hello-webapp/target/repository/`

This is the content that contains:

- `site.xml`
- `artifacts.jar`
- `content.jar`
- `features/`
- `plugins/`

This is the same structure you previously imported manually from your local machine.

## Recommended Deployment Flow

Use this sequence:

1. wait for the GitHub Actions run to complete successfully
2. download the `update-site-repository` artifact
3. extract it locally
4. locate `site.xml`
5. open `updatesite.nsf`
6. import the extracted update site
7. remove older versions if needed
8. restart Domino HTTP
9. verify the deployed URLs

## Option A: Download Artifact In GitHub Web UI

This is the simplest path.

1. Open the repository in GitHub
2. Open the `Actions` tab
3. Open the latest successful run of `Build Tutorial Site`
4. Scroll to the `Artifacts` section
5. Download `update-site-repository`
6. Extract the downloaded archive
7. Open the extracted folder and confirm it contains `site.xml`

Then import that `site.xml` into `updatesite.nsf`.

## Option B: Download Artifact With GitHub CLI

If you prefer CLI:

```powershell
gh run download 24794809561 `
  --repo bryanHsiao/domino-static-webapp-tutorial `
  --name update-site-repository `
  --dir artifacts/update-site-repository
```

After that, the extracted p2 repository should be in:

- `artifacts/update-site-repository/`

You should then find:

- `artifacts/update-site-repository/site.xml`

That is the file to import into `updatesite.nsf`.

Note:

- replace `24794809561` with the latest successful run id when needed

## How To Find The Latest Successful Run ID

Using GitHub CLI:

```powershell
gh run list `
  --repo bryanHsiao/domino-static-webapp-tutorial `
  --workflow ci.yml `
  --limit 5
```

Look for the latest run with:

- status: `completed`
- conclusion: `success`

Then use that run id in `gh run download`.

## Importing Into `updatesite.nsf`

Once the artifact is extracted:

1. open the Domino update site NSF
2. choose `Import Local Update Site`
3. select the extracted `site.xml`
4. wait for the import to finish

This updates the feature and plugin entries contained in the update site NSF.

## Handling Existing Versions

If the NSF already contains older imported versions of the same feature/plugin, you have two reasonable options:

### Option 1: Keep only the newest version

Recommended for clarity.

- remove the older `hello-webapp` feature/plugin entries
- keep only the latest imported version

### Option 2: Keep history

Useful for tutorial comparison, but less clean operationally.

- retain old versions in the NSF
- ensure you understand which version Domino is actually loading

Operationally, Option 1 is usually better.

## Restarting Domino HTTP

After the import, restart the HTTP task:

```text
restart task http
```

This allows Domino to reload the bundles from `OSGI_HTTP_DYNAMIC_BUNDLES`.

## Recommended Verification URLs

After the restart, verify:

- `/hello-webapp/`
- `/hello-webapp/docs/index.html`
- `/hello-webapp/step1/index.html`
- `/hello-webapp/step2/index.html`
- `/hello-webapp/step3/index.html`

If Step 3 is part of the deployment, also verify:

- the page loads correctly
- the live `/api` request succeeds
- the cards and JSON panel populate

## Suggested Team Workflow

For practical day-to-day usage:

1. developers push to GitHub
2. GitHub Actions builds and publishes artifacts
3. deployer downloads `update-site-repository`
4. deployer imports into `updatesite.nsf`
5. deployer restarts HTTP
6. deployer validates the tutorial site

This gives you repeatable CI without needing fully automated Domino-side CD yet.

## Future Improvement Path

If you later want to reduce manual work further, possible next steps are:

1. add a release workflow that attaches artifacts to GitHub Releases
2. store artifact metadata and deployment notes automatically
3. design a controlled CD path for Domino update-site import
4. automate post-deploy verification

## 如何匯入 `updatesite.nsf`

artifact 解壓之後：

1. 打開 Domino 的 update site NSF
2. 選 `Import Local Update Site`
3. 指到解壓後的 `site.xml`
4. 等匯入完成

這一步會更新 NSF 中的 feature 與 plugin 資料。

## 如何處理舊版本

如果 NSF 裡已經有同一組 feature/plugin 的舊版本，你有兩種做法：

### 做法 1：只保留最新版本

這是比較建議的做法。

- 把舊的 `hello-webapp` feature/plugin 刪掉
- 只保留最新匯入版本

### 做法 2：保留歷史版本

如果你是教學用途，也可以保留。

- 保留舊版 entries
- 但你要清楚 Domino 實際載入的是哪個版本

就實務維運來說，通常做法 1 比較乾淨。

## 匯入後重啟 Domino HTTP

匯入之後，請重啟 HTTP：

```text
restart task http
```

這樣 Domino 才會重新載入 `OSGI_HTTP_DYNAMIC_BUNDLES` 裡的 bundle。
